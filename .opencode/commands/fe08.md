# /fe08 — Application Review Detail (Staff)

## What you are building in this sprint

The most complex staff-facing page: the application review detail. Officers click into
an application from the queue and need to see everything — form data, documents,
document controls, and decision controls — in one view.

## Dependency check

FE-07 (StaffShell, Table, portal queue). Backend: all review endpoints confirmed working
from sprint 8 backend tests.

---

## Deliverables

### 1. Application Review Detail — `frontend/app/portal/applications/[id]/page.tsx`

Client component. Fetches:
- `GET /api/v1/applications/{id}/` → full application + form_data + documents
- `GET /api/v1/applications/{id}/history/` → status history

**Page header:**
`<PageHeader title={applicant_name} breadcrumb={[{ label: 'Applications', href: '/portal/applications' }, { label: applicant_name }]} />`

**Two-column layout on lg+, stacked on mobile:**
- Left column (wider, ~65%): applicant info + documents
- Right column (narrower, ~35%, sticky): decision panel

---

**Left column — Applicant Information card:**
`<Card>` with section heading "Applicant Information"

Renders `form_data` as a definition list (`<dl>`):
- Full Name
- Phone Number
- Nationality
- Date of Birth
- Personal Statement (full text, not truncated)

Also show: Applicant email (`text-sm text-text-400`)

---

**Left column — Documents card:**
`<Card>` with section heading "Required Documents"

Each required document as a row:
```
[FileIcon]  Document Type Label       [StatusBadge]
            Last updated date         [Verify] [Flag] buttons
```

**Verify button:**
- `<Button variant="secondary" size="sm">Verify</Button>`
- Only shown if status is `pending`
- On click: `PATCH /api/v1/documents/{id}/verify/`
- On success: status badge updates to verified (optimistic update)
- On error: `<Alert variant="danger">` inline

**Flag button:**
- `<Button variant="danger" size="sm">Flag</Button>`
- Only shown if status is `pending`
- On click: opens inline flag reason form:
  ```
  <Textarea placeholder="Describe the issue with this document..." rows={3} />
  <Button variant="danger" size="sm">Confirm Flag</Button>
  <Button variant="ghost" size="sm">Cancel</Button>
  ```
- API: `PATCH /api/v1/documents/{id}/flag/` with `{ reason: "..." }`
- Reason is required — disable "Confirm Flag" if reason is empty
- On success: status updates to flagged, reason shown below document row
- On error: `<Alert variant="danger">` inline

**Verified document:** show `<StatusBadge status="accepted">` (green "Verified"),
reviewer name + date in `text-xs text-text-400`.

**Flagged document:** show `<StatusBadge status="rejected">` (red "Flagged"),
flagged reason in `text-sm text-danger italic` below.

---

**Left column — Status Timeline card:**
`<Card>` with section heading "Application Timeline"

Vertical timeline from `history` endpoint. Same component pattern as FE-05 applicant
timeline but rendered here for staff context. Most recent at top.

---

**Right column — Decision Panel (sticky):**
`<Card>` with section heading "Decision"

**State: draft or submitted (not yet under review):**
```
<Alert variant="info">
  This application has not been opened for review yet.
</Alert>
<Button variant="primary" className="w-full mt-3">Open for Review</Button>
```
"Open for Review" calls `PATCH /applications/{id}/status/` with `{ status: 'under_review' }`.

**State: under_review:**

Document readiness check (client-side):
- Count verified documents vs total required
- If not all verified: show warning and disable decision buttons
  ```
  <Alert variant="warning">
    {unverified} document(s) need verification before a decision can be issued.
  </Alert>
  ```
- If all verified: show decision buttons

Decision buttons (when all docs verified):
```
<Button variant="primary" className="w-full">Admit</Button>
<Button variant="secondary" className="w-full mt-2">Waitlist</Button>
<Button variant="danger" className="w-full mt-2">Reject</Button>
```

Each opens a confirmation modal:
```
<Modal title="Confirm Decision">
  <p>Are you sure you want to mark this application as [decision]?</p>
  <p className="text-sm text-text-400 mt-1">This action can be reversed if the applicant has not yet responded.</p>
  [Cancel] [Confirm]
</Modal>
```

API: `PATCH /api/v1/applications/{id}/status/` with `{ status: 'admitted' | 'rejected' | 'waitlisted' }`

**State: admitted/rejected/waitlisted (decision issued, no applicant response):**
```
<Alert variant={status === 'admitted' ? 'success' : 'danger'}>
  Decision issued: {status} on {decision_at}
</Alert>
<Button variant="ghost" className="w-full mt-3">Reverse Decision</Button>
```

"Reverse Decision" opens a modal with a required reason textarea:
```
<Modal title="Reverse Decision">
  <p>Provide a reason for reversing this decision:</p>
  <Textarea placeholder="Reason for reversal..." rows={3} required />
  [Cancel] [Confirm Reversal]
</Modal>
```
API: `PATCH /applications/{id}/status/` with `{ status: 'under_review', reason: '...' }`

**State: accepted/declined (applicant responded):**
```
<Alert variant="info">
  Applicant has {status} the offer on {offer_response_at}.
  No further action required.
</Alert>
```
No decision buttons shown. Decision cannot be reversed after applicant responds.
