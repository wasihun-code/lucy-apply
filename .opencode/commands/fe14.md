# /fe14 — MFA Pages Redesign

## What you are building in this sprint

MFA setup and verify pages were built functionally in backend sprint 10 but with
minimal styling. This sprint aligns them with the design system. No logic changes —
visual redesign only.

## Dependency check

FE-04 (AuthCard component). MFA pages must already be functional (sprint 10 backend).

## Deliverables

### 1. MFA Setup — `frontend/app/(auth)/mfa/setup/page.tsx`

Currently functional but unstyled. Redesign using `AuthCard`.

**Keep:** all existing logic — API call to `POST /auth/mfa/setup/`, QR code display,
OTP verification step.

**Redesign:**

Inside `<AuthCard>`:

**Step 1: QR Code display**
```
<ShieldCheck size={40} className="text-primary mx-auto mb-4" />
<h1 className="text-2xl font-display font-bold text-center">
  Set up two-factor authentication
</h1>
<p className="text-sm text-text-600 text-center mt-2">
  Scan this QR code with your authenticator app (Google Authenticator, Authy, etc.)
</p>

[QR Code image — center aligned, 200×200, white bg, rounded-lg, shadow-sm]

<details className="mt-4">
  <summary className="text-sm text-primary cursor-pointer">
    Can't scan? Enter code manually
  </summary>
  <p className="font-mono text-sm bg-background p-3 rounded mt-2 text-center select-all">
    {secret_key}
  </p>
</details>
```

**Step 2: Verify the setup** (same page, after showing QR):
```
<p className="text-sm text-text-600 mt-6">
  Enter the 6-digit code from your app to confirm setup:
</p>
<Input
  type="text"
  inputMode="numeric"
  maxLength={6}
  placeholder="000000"
  className="text-center text-2xl tracking-[0.5em] font-mono"
/>
<Button variant="primary" size="lg" className="w-full mt-4">
  Verify & Enable
</Button>
```

Error: `<Alert variant="danger">{error}</Alert>` above the input.
Success: redirect to the appropriate portal/admin page.

**Remaining attempts display:**
If `remaining_attempts` is returned on failure, show:
`<p className="text-xs text-warning mt-1">{remaining_attempts} attempts remaining</p>`

### 2. MFA Verify — `frontend/app/(auth)/mfa/verify/page.tsx`

Shown after login when MFA is enabled. Keep all logic, redesign visual.

Inside `<AuthCard>`:

```
<Shield size={40} className="text-primary mx-auto mb-4" />
<h1 className="text-2xl font-display font-bold text-center">
  Two-factor authentication
</h1>
<p className="text-sm text-text-600 text-center mt-2">
  Enter the 6-digit code from your authenticator app.
</p>

<Input
  type="text"
  inputMode="numeric"
  maxLength={6}
  placeholder="000000"
  className="text-center text-2xl tracking-[0.5em] font-mono mt-6"
  autoFocus
/>

{error && <Alert variant="danger" className="mt-3">{error}</Alert>}
{remaining_attempts && (
  <p className="text-xs text-warning mt-1 text-center">
    {remaining_attempts} attempts remaining before lockout
  </p>
)}

<Button variant="primary" size="lg" className="w-full mt-4" loading={isLoading}>
  Verify
</Button>

<Button variant="ghost" size="sm" className="w-full mt-2" onClick={handleLogout}>
  Sign in with a different account
</Button>
```

**Lockout state** (when remaining_attempts === 0 or 429 returned):
```
<Alert variant="danger">
  Too many incorrect attempts. Please wait 5 minutes before trying again.
</Alert>
```
Hide the code input and verify button. Show only the "Sign in with different account" button.

### 3. MFA Redirect Logic in Login

Verify the login page (FE-04) correctly redirects to MFA pages:
- Staff/admin with `mfa_enabled === false` → `/mfa/setup`

If this redirect logic is already correct from FE-04, just verify it — no changes needed.
