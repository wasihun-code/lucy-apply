# /fe16 — Status Timeline + Notifications Inbox

## What you are building in this sprint

Two polish features that significantly improve the applicant experience: a proper
application status timeline component (reused across applicant and staff views) and
a basic in-app notifications inbox.

## Dependency check

FE-05 (ApplicationCard, dashboard), FE-08 (timeline used in review detail).
Backend: `/applications/{id}/history/` confirmed working.

---

## Deliverables

### 1. StatusTimeline Component — `frontend/components/shared/StatusTimeline.tsx`

A reusable vertical timeline used in:
- Applicant application detail (`/dashboard/applications/[id]`)
- Staff application review detail (`/portal/applications/[id]`)

Currently both pages fetch history and render it ad-hoc. Extract into a shared component.

```tsx
type TimelineEntry = {
  id: string
  from_status: string | null
  to_status: string
  changed_by_type: 'applicant' | 'university_staff' | 'system'
  reason?: string
  created_at: string
}

type StatusTimelineProps = {
  entries: TimelineEntry[]
  loading?: boolean
}
```

Layout — vertical timeline (most recent first):

```
[●] Admitted                           Jun 25, 2026
    Decision issued by admissions team
    ← Previously: Under Review

[●] Under Review                       Jun 24, 2026
    Application opened for review

[●] Submitted                          Jun 23, 2026
    Payment received · Application submitted
```

Visual details:
- Each node: circle `●` in status semantic color (use color tokens matching StatusBadge)
- Connecting vertical line: `bg-border` 2px wide
- Status label: `text-sm font-semibold text-text-900` with `<StatusBadge>` inline
- Date: `text-xs text-text-400` right-aligned
- Description line: `text-sm text-text-600`
- Reason (if present): `text-sm text-text-400 italic`
- From-status: `text-xs text-text-400` "← Previously: {from_status}"

Actor labels:
- `applicant` → "by applicant"
- `university_staff` → "by admissions team"
- `system` → "automatically"

Loading state: 3× skeleton entries (circle + two lines each).

**Update both detail pages** to use this shared component instead of any ad-hoc timeline.

### 2. Notifications Inbox — `frontend/app/dashboard/notifications/page.tsx`

Lucy Apply sends emails for all applicant-facing events (backend sprint 5+).
There is no in-app notification system yet. This sprint adds a basic inbox that
reads from the application status history to surface actionable events.

**Important scope note:** There is no dedicated notifications backend endpoint.
Build this as a derived view from the applicant's applications data. Show events
that are relevant to the applicant (status changes, document flags, decisions).

Client component. Fetches:
- `GET /api/v1/applicants/me/applications/` → all applications
- For each application (max 5 most recent): `GET /api/v1/applications/{id}/history/`

Derive notification items from history entries:
- `to_status === 'under_review'` → "Your application is being reviewed"
- `to_status === 'admitted'` → "You've been admitted! 🎉" (accent color, high priority)
- `to_status === 'rejected'` → "Application decision received"
- `to_status === 'waitlisted'` → "You've been waitlisted"
- Document flagged events → "A document needs your attention"

Sort all derived notifications by date, newest first.

**Page header:** `<PageHeader title="Notifications" />`

Add "Notifications" to ApplicantShell sidebar nav.

**Notification list:**

Each notification as a list row:

```
[●] Status icon (colored per type)    Application to {program.name}
                                      {notification message}
                                      {relative time: "2 days ago"}
```

High-priority items (admitted, flagged): slightly elevated card, `border-l-4 border-accent` or `border-warning`.

On click: navigate to `/dashboard/applications/{id}` (the relevant application).

**Empty state:**
`<EmptyState icon={<Bell>} heading="No notifications" description="Updates about your applications will appear here." />`

**Loading:** 5× skeleton rows.

**Notification bell in ApplicantShell top bar:**
Add a bell icon `<Bell>` to the top bar of ApplicantShell.
Show a badge count if there are any "unread" notifications (naive: count admitted/flagged
events in the last 7 days as "unread" until the user visits the page).
Store "last visited notifications" in `localStorage` to compute unread count.
Badge: `bg-danger text-white text-xs rounded-full h-4 w-4` positioned top-right of bell.
