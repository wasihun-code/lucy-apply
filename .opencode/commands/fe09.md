# /fe09 — Programs Management + Cycle Management

## What you are building in this sprint

The program management experience for university admins — creating, editing, and
publishing programs — and the admission cycle management page, which currently
has no UI at all.

## Dependency check

FE-07 (StaffShell wired). Backend: program and cycle endpoints all working from sprint 7.
Admin-only actions require `permission_level === 'admin'` check.

---

## Deliverables

### 1. Programs List — `frontend/app/portal/programs/page.tsx`

**Keep:** existing list. **Redesign:** visual presentation + add program status filter.

Client component. Fetches `GET /api/v1/universities/{id}/programs/`.

**Page header:**
`<PageHeader title="Programs" action={isAdmin && <Button variant="primary" size="sm" href="/portal/programs/new">+ New Program</Button>} />`

"+ New Program" only shown if `permission_level === 'admin'`.

**Filter bar:** Status dropdown: All | Draft | Published | Archived

**Program list:** Vertical list of program rows (not cards):

Each row:
```
Program Name (text-base font-semibold)        [StatusBadge]    [Cycles] [Edit]
degree_level · USD fee_amount                                   (admin only)
```

- `[Cycles]` button: `<Button variant="secondary" size="sm">Cycles</Button>` → `/portal/programs/{id}/cycles`
- `[Edit]` button: `<Button variant="ghost" size="sm"><Pencil size={16} /></Button>` → `/portal/programs/{id}/edit`
- Edit and status actions only shown to admins
- Published badge: green. Draft: gray. Archived: gray + strikethrough on name.

**Empty state:** `<EmptyState icon={<BookOpen>} heading="No programs yet" description="Create your first program to start accepting applications." action={{ label: '+ New Program', href: '/portal/programs/new' }} />`
(Action only shown to admins.)

### 2. New Program Form — `frontend/app/portal/programs/new/page.tsx`

Admin only. Redirect to `/portal/programs` if `permission_level !== 'admin'`.

`<PageHeader title="New Program" breadcrumb={[{ label: 'Programs', href: '/portal/programs' }, { label: 'New' }]} />`

Form inside `<Card padding="lg">`:

```
Section: "Program Details"
  Name *                    <Input>
  Degree Level *            <Select> — undergraduate | postgraduate
  Description               <Textarea rows={4}>
  Requirements              <Textarea rows={4} hint="Entry requirements, qualifications needed">

Section: "Application Fee"
  Fee Amount *              <Input type="number" min="0" step="0.01">
  Currency                  <Select> — USD (only option for now) | placeholder for future

Section: "Required Documents"
  Dynamic list of document type + label pairs
  [ + Add Document ] button adds a new row
  Each row: <Input placeholder="Type (e.g. transcript)"> + <Input placeholder="Label"> + [×] remove
  At least one document required

Footer buttons:
  <Button variant="secondary">Save as Draft</Button>
  <Button variant="primary">Save & Publish</Button>
```

API: `POST /api/v1/universities/{id}/programs/` with body:
```json
{
  "name": "...",
  "degree_level": "undergraduate",
  "description": "...",
  "requirements": "...",
  "fee_amount": "50.00",
  "fee_currency": "USD",
  "required_documents": [{ "type": "transcript", "label": "Official Transcript" }],
  "status": "draft" | "published"
}
```

On success: redirect to `/portal/programs`.
On error: `<Alert variant="danger">{error}</Alert>` above form.

### 3. Edit Program Form — `frontend/app/portal/programs/[id]/edit/page.tsx`

Same form as New Program, pre-populated with existing data.
Fetches `GET /api/v1/programs/{id}/` on load.

Additional controls:
- If status is `published`: show `<Button variant="secondary">Archive Program</Button>`
  with confirmation modal. API: `PATCH /programs/{id}/status/` with `{ status: 'archived' }`
- If status is `draft`: show `<Button variant="primary">Publish Program</Button>`
  API: `PATCH /programs/{id}/status/` with `{ status: 'published' }`
- If status is `archived`: show `<Alert variant="info">This program is archived.</Alert>`
  — no publish/archive button (archived is terminal)

API: `PATCH /api/v1/programs/{id}/` with changed fields.

### 4. Cycle Management — `frontend/app/portal/programs/[id]/cycles/page.tsx`

New page. Client component.
Fetches: `GET /api/v1/programs/{id}/cycles/`

`<PageHeader title="Admission Cycles" breadcrumb={[{ label: 'Programs', href: '/portal/programs' }, { label: program.name, href: '/portal/programs/{id}/edit' }, { label: 'Cycles' }]} action={isAdmin && <Button variant="primary" size="sm">+ New Cycle</Button>} />`

**Cycle list:** `<Table>` with columns:

| Column | Content |
|---|---|
| Cycle Name | Name text |
| Opens | open_date formatted |
| Closes | close_date formatted |
| Status | `<StatusBadge>` |
| Actions | Close / Archive buttons (admin only, status-dependent) |

**Status-dependent actions:**
- `scheduled`: no action (Celery will auto-open)
- `open`: `<Button variant="danger" size="sm">Close Early</Button>` with confirmation modal
- `closed`: `<Button variant="secondary" size="sm">Archive</Button>` with confirmation modal
- `archived`: no action

Close early API: `PATCH /api/v1/admission-cycles/{id}/close/`
Archive API: `PATCH /api/v1/admission-cycles/{id}/archive/`

**+ New Cycle modal/form** (opens in modal, not new page — simpler for this):

```
<Modal title="New Admission Cycle">
  Cycle Name *      <Input>
  Open Date *       <Input type="datetime-local">
  Close Date *      <Input type="datetime-local">
  [Cancel] [Create Cycle]
</Modal>
```

Client-side validation: close_date must be after open_date.
API: `POST /api/v1/programs/{id}/cycles/`

**Empty state:** `<EmptyState icon={<Calendar>} heading="No admission cycles" description="Create a cycle to start accepting applications for this program." />`
