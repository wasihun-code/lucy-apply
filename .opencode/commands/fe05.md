# /fe05 — Applicant Dashboard + Application Detail

## What you are building in this sprint

The applicant's home base after login. Currently a near-empty page showing name, email,
and a flat list of application cards. This sprint transforms it into a proper dashboard
and adds the missing application detail page applicants need to view submitted applications.

## Dependency check

FE-01 (shells + components), FE-04 (auth). `ApplicantShell` must be wired and working.

---

## Deliverables

### 1. Applicant Dashboard — `frontend/app/dashboard/page.tsx`

Client component. Fetches:
- `GET /api/v1/auth/me/` → user info
- `GET /api/v1/applicants/me/applications/` → applications list

**Layout sections:**

**Greeting header:**
- "Welcome back, {first_name}" (`text-2xl font-display font-bold`)
- Date/time subtext: `text-sm text-text-400`
- No more "Name: Bruno / Email: applicant2@test.com" label dump

**Summary stat cards row** (4 cards, horizontal scroll on mobile):
- Total Applications
- Pending Review (under_review count)
- Decisions (admitted + rejected + waitlisted count)
- Accepted (accepted count)
Each card: `<Card>` with a number (`text-3xl font-bold`) + label + lucide icon.
Counts computed client-side from the applications list.

**Applications section:**
- Heading: "My Applications" + `<Button variant="primary" size="sm">Browse Programs</Button>` right-aligned
- List of `<ApplicationCard>` components (see below)
- Empty state: `<EmptyState icon={<FileText>} heading="No applications yet" description="Browse our partner universities to find the right program." action={{ label: 'Browse Programs', href: '/universities' }} />`
- Loading: 3× `<SkeletonCard />`

**ApplicationCard Component** — `frontend/components/shared/ApplicationCard.tsx`

```tsx
type ApplicationCardProps = {
  application: {
    id: string
    status: string
    submitted_at: string | null
    program: { name: string; university: { name: string } }
    document_checklist?: { total: number; verified: number }
  }
}
```

Layout (list row style, not grid card):
- Left: program name (`text-base font-semibold`) + university name (`text-sm text-text-400`)
- Center: document progress "2/3 verified" (`text-sm text-text-600`) — hide for draft
- Right: `<StatusBadge status={application.status} />` + submitted date if present
- Full row is clickable → `/dashboard/applications/{id}`
- Hover: `bg-primary-soft/30` row highlight

Status-based visual treatment:
- `admitted`: left border `border-l-4 border-accent` (gold — achievement moment)
- `accepted`: left border `border-l-4 border-success`
- `rejected`: left border `border-l-4 border-danger`
- `draft`: no border accent
- others: `border-l-4 border-primary/20`

### 2. Application Detail — `frontend/app/dashboard/applications/[id]/page.tsx`

**NEW PAGE** — fixes BUG-FE-003. Client component.

Fetches:
- `GET /api/v1/applications/{id}/` → full application
- `GET /api/v1/applications/{id}/history/` → status history

This is a READ-ONLY view. Applicants cannot edit a submitted application.

**Page header:**
`<PageHeader title={program.name} breadcrumb={[{ label: 'Dashboard', href: '/dashboard' }, { label: program.name }]} />`

**Status banner** (prominent, top of content):
```
<Card> with left border matching status color
  <StatusBadge status={application.status} large />
  Status label text (human readable)
  Submitted date if present
```

Special cases:
- If `admitted` and `offer_response_at` is null: show offer response UI (see below)
- If `accepted` or `declined`: show the response with date

**Offer Response UI** (only when status === 'admitted' and offer_response_at is null):
```
<Card className="border-accent border-2">
  Heading: "You've been admitted! 🎉" (accent color)
  Body: "Please respond to your offer by reviewing the information below."
  Two buttons side by side:
  - <Button variant="primary">Accept Offer</Button>
  - <Button variant="secondary">Decline Offer</Button>
  Confirmation modal before submitting either action.
</Card>
```

API: `POST /api/v1/applications/{id}/offer-response/` with `{ response: 'accepted' | 'declined' }`

**Application Information section:**
- Card with `form_data` fields rendered as key-value pairs
- Use a `<DefinitionList>` pattern: `<dl>` with `<dt>` label + `<dd>` value
- Only show fields that have values

**Documents section:**
- Card with document checklist
- Each document: icon + document type label + `<StatusBadge status={doc.status} />`
- If `flagged`: show flagged reason in `text-sm text-danger` below
- If `flagged`: show "Re-upload" button → navigates back to wizard step 2

**Status History / Timeline:**
- Heading: "Application Timeline"
- Vertical timeline: each `ApplicationStatusHistory` entry as a timeline node
- Node: date + "Status changed to {to_status}" + reason if present
- Most recent at top
- Use `<StatusBadge>` inline within timeline items
- Loading: skeleton timeline
