# /fe04 — Auth Pages

## What you are building in this sprint

All authentication pages redesigned using the design system. Login, register, forgot
password, reset password, and email verification. These are the first pages new users
see — they must feel polished and trustworthy.

## Dependency check

FE-01 must be complete. All form primitives (Input, Button, FormField, Alert) must exist.

New page (currently missing). Reads `?token=` from URL query params.

Layout inside `<AuthCard>`:
- Heading: "Set new password"
- If no token in URL: show `<Alert variant="danger">Invalid or expired reset link.</Alert>`
- Fields:
  - New Password (password, required) — hint: "Minimum 8 characters"
  - Confirm Password (password, required)

### 5. Verify Email — `frontend/app/(auth)/verify-email/page.tsx`

New page (currently missing). Reads `?token=` from URL query params.
This is a landing page — user clicks the link from their email.

On mount: immediately call `POST /api/v1/auth/verify-email/` with the token.

States:
- **Loading:** `<Skeleton className="h-32 w-full" />` inside AuthCard
- **Success:**
  ```
  ✓ icon (CheckCircle from lucide, text-success, 48px)
  Heading: "Email verified!"
  Body: "Your account is ready. You can now sign in and start applying."
  Button: "Sign In" → /login
  ```
- **Error (invalid/expired token):**
  ```
  ✗ icon (XCircle from lucide, text-danger, 48px)
  Heading: "Verification failed"
  Body: "This link may have expired. Request a new one."
  Button: "Resend verification email" — calls POST /auth/resend-verification/
  ```

This fixes the missing page that previously caused a 404 after email signup.
